## 개요

앞 글에서는 DMA가 DDR과 AXI-Stream 사이에서 데이터를 옮기는 역할을 한다고 정리했다.
하지만 DMA와 Systolic Array Engine을 실제로 사용하려면 누군가가 설정값을 써줘야 한다.

예를 들어 software는 다음 일을 해야 한다.

```text
Activation DMA source address 설정
Weight DMA source address 설정
Result DMA destination address 설정
DMA transfer length 설정
Systolic Array Engine의 M/N/K size 설정
BRAM base address 설정
start bit write
done/busy 상태 read
```

이때 PS와 PL 사이에서 register를 읽고 쓰는 통로가 **AXI4-Lite**다.

Zynq 계열 FPGA에서는 ARM processor가 있는 쪽을 PS(Processing System), FPGA logic이 있는 쪽을 PL(Programmable Logic)이라고 부른다.
AXI4-Lite는 PS가 PL 내부 IP의 register에 접근할 수 있게 해주는 memory-mapped control bus다.

한 문장으로 정리하면 다음과 같다.

```text
AXI4-Lite = PS가 PL의 control/status register를 읽고 쓰는 통로
```

## AXI4-Lite와 DMA의 차이

AXI4-Lite와 DMA를 같은 "통신"으로 묶어서 생각하면 헷갈리기 쉽다.
둘은 역할이 다르다.

| 구분 | 역할 | 데이터 크기 | 예시 |
|---|---|---:|---|
| AXI4-Lite | register 제어 | 보통 32bit 단위 | start, done, address, length 설정 |
| AXI DMA | 대량 데이터 이동 | 수 KB ~ MB 이상 | DDR의 matrix buffer를 BRAM으로 이동 |
| AXI-Stream | 연속 datapath | beat 단위 | DMA와 adapter 사이의 data stream |

AXI4-Lite는 빠른 대량 데이터 전송용 버스가 아니다.
burst도 없고, 한 번에 register 하나를 읽거나 쓰는 단순한 bus다.
대신 구조가 단순해서 control register를 만들기에 좋다.

이번 Systolic Array 시스템에서는 다음처럼 나눠서 생각한다.

```text
Control path:
  PS -> AXI4-Lite -> DMA register / Systolic Array control register

Data path:
  DDR -> DMA -> AXI-Stream -> BRAM
  BRAM -> AXI-Stream -> DMA -> DDR
```

즉 PS가 AXI4-Lite로 직접 matrix 데이터를 한 word씩 쓰는 것이 아니다.
PS는 AXI4-Lite로 DMA와 Engine의 register만 설정하고, 실제 큰 데이터 이동은 DMA가 처리한다.

## Vivado에서의 연결 방향

AXI4-Lite는 memory-mapped bus다.
PS의 processor가 특정 주소에 write하면, 그 주소가 PL 쪽 IP의 register로 decode된다.

Vivado Block Design에서는 보통 다음처럼 연결한다.

```text
PS M_AXI_HPM
  -> AXI SmartConnect / AXI Interconnect
      -> Activation DMA S_AXI_LITE
      -> Weight DMA S_AXI_LITE
      -> Result DMA S_AXI_LITE
      -> Systolic Array Controller S_AXI_LITE
```

즉 PS의 AXI master port 하나가 SmartConnect를 통해 여러 AXI4-Lite slave register block에 접근하는 구조다.

여기서 master/slave 방향이 중요하다.

| 포트 | 역할 |
|---|---|
| PS `M_AXI_HPM*` | PS가 AXI transaction을 만드는 master |
| DMA `S_AXI_LITE` | DMA register를 제공하는 slave |
| Custom Controller `S_AXI_LITE` | Engine control/status register를 제공하는 slave |

앞 글에서 DMA의 data port는 반대 방향이었다.
DMA의 `M_AXI_MM2S`, `M_AXI_S2MM`은 DMA가 master가 되어 PS DDR 쪽 slave port로 접근한다.

정리하면 다음과 같다.

```text
제어 경로:
  PS master -> PL AXI4-Lite slave register

데이터 경로:
  DMA master -> PS DDR slave port
```

이 두 방향을 구분해야 Block Design을 연결할 때 덜 헷갈린다.

## Vivado에서 AXI4-Lite IP 생성하기

AXI4-Lite slave 코드는 처음부터 직접 작성할 수도 있지만, 처음에는 Vivado가 만들어주는 template을 사용하는 편이 안전하다.
Vivado에서는 다음 메뉴에서 AXI4-Lite peripheral을 생성할 수 있다.

```text
Tools
  -> Create and Package New IP
  -> Create a new AXI4 peripheral
```

첫 화면에서는 `Create a new AXI4 peripheral`을 선택한다.
이 옵션은 AXI4 IP wrapper, driver, test application, simulation 예제를 같이 생성해준다.

![](/assets/images/axi4lite_ip_create.png)

다음 화면에서는 IP 이름과 저장 위치를 정한다.
예시에서는 이름을 `axi4lite`, version을 `1.0`, IP 저장 위치를 `ip_repo`로 두었다.

![](/assets/images/axi4lite_ip_details.png)

이름은 나중에 Vivado IP Catalog에 보이는 이름이므로, 실제 프로젝트에서는 `systolic_ctrl`, `sa_controller`, `axi4lite_ctrl`처럼 역할이 드러나게 짓는 편이 좋다.
`IP location`은 Vivado project 안쪽이나 별도 `ip_repo` directory로 잡을 수 있다.

다음은 interface 설정이다.

![](/assets/images/axi4lite_ip_interface.png)

여기서는 AXI4-Lite slave register block을 만들 것이므로 다음처럼 둔다.

| 항목 | 설정 | 의미 |
|---|---|---|
| Interface Type | `Lite` | AXI4-Lite register interface |
| Interface Mode | `Slave` | PS가 master, 이 IP가 slave |
| Data Width | `32` | register 하나가 32bit |
| Number of Registers | `4` | `slv_reg0` ~ `slv_reg3` 생성 |
| Enable Interrupt Support | off | 처음에는 polling 방식으로 단순화 |

`Number of Registers = 4`로 만들면 address space는 64byte로 잡히고, template 안에는 4개의 32bit register가 생긴다.
offset으로 보면 보통 다음처럼 사용한다.

| Register | Offset |
|---|---:|
| `slv_reg0` | `0x00` |
| `slv_reg1` | `0x04` |
| `slv_reg2` | `0x08` |
| `slv_reg3` | `0x0C` |

이 4개 register는 AXI4-Lite 동작을 이해하기 위한 최소 예시로 충분하다.
하지만 앞에서 잡은 Systolic Array Controller register map은 `CTRL`, `M_SIZE`, `N_SIZE`, `K_SIZE`, `ACT_BASE`, `WEIGHT_BASE`, `ACC_BASE`, `STATUS`까지 필요하므로 4개보다 많다.
실제 controller용 IP를 만들 때는 `Number of Registers`를 8개 또는 16개로 늘려 생성하거나, 생성된 template을 확장하면 된다.

마지막 summary 화면에서는 생성될 peripheral 정보를 확인한다.
여기서는 AXI4-Lite interface가 1개인 `axi4lite_v1.0` IP가 만들어지고, 생성 후 바로 수정할 수 있도록 `Edit IP`를 선택했다.

![](/assets/images/axi4lite_ip_summary.png)

`Finish`를 누르면 Vivado가 AXI4-Lite slave template을 생성한다.
첨부한 코드의 module 이름은 다음 형태다.

```verilog
module axi4lite_slave_lite_v1_0_S00_AXI #(
    parameter integer C_S_AXI_DATA_WIDTH = 32,
    parameter integer C_S_AXI_ADDR_WIDTH = 4
) (
    input wire S_AXI_ACLK,
    input wire S_AXI_ARESETN,
    ...
);
```

이 template에는 AXI4-Lite handshake를 처리하는 기본 FSM과 4개의 예제 register가 들어 있다.

```verilog
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg0;
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg1;
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg2;
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg3;
```

처음 봐야 할 부분은 AXI 신호 전체보다 `slv_reg0`~`slv_reg3`가 어디에서 write되고, read data로 어떻게 선택되는지다.
PS에서 `MMIO.write(base + 0x04, value)`를 하면 결국 `slv_reg1`이 갱신되는 식이다.

따라서 우리가 할 일은 template의 AXI handshake 부분을 크게 건드리는 것이 아니라, user logic register 부분을 Systolic Array Controller에 맞게 바꾸는 것이다.

```text
slv_reg0[0] -> start pulse 생성
slv_reg1    -> m_size
slv_reg2    -> n_size
slv_reg3    -> k_size
...
done/busy/error -> read-only status register로 연결
```

Vivado가 생성한 template은 학습용으로는 편하지만, 실제 accelerator control block으로 쓰려면 register 수, read-only/write-only 동작, write-one-to-clear bit 같은 정책을 직접 정리해야 한다.

이 시리즈에서는 첨부한 template을 바탕으로 submodule에 `external/SimpleSystolicArray/hdl/axi4lite_slave_lite_v1_0_S00_AXI.sv`를 추가했다.
현재 버전은 4개 register를 다음처럼 사용한다.

| Offset | Write 동작 | Read 동작 |
|---:|---|---|
| `0x00` | bit0 write 시 `start_o` 1 cycle pulse, bit1 write 시 done clear | bit0 `done`, bit1 `busy` |
| `0x04` | `m_size_o` 설정 | `m_size_o` read |
| `0x08` | `n_size_o` 설정 | `n_size_o` read |
| `0x0C` | `k_size_o` 설정 | `k_size_o` read |

Engine 쪽으로 나가는 신호는 다음과 같다.

```verilog
output logic [31:0] m_size_o,
output logic [31:0] n_size_o,
output logic [31:0] k_size_o,
output logic        start_o,
input  logic        done_i,
input  logic        busy_i
```

`start_o`는 software가 `0x00` register의 bit0에 1을 쓸 때 한 cycle만 올라간다.
`done_i`는 Engine에서 들어오는 완료 pulse이고, AXI4-Lite slave 안에서 sticky bit로 잡아둔다.
software가 `0x00` register의 bit1에 1을 쓰면 이 done bit를 clear한다.

이 동작은 `external/SimpleSystolicArray/tb/tb_axi4lite_slave_lite.sv`로 검증했다.
테스트에서는 M/N/K write와 readback, start pulse, busy status, done latch/clear를 확인한다.

```text
tb_axi4lite_slave_lite PASS
```

## Address Map

AXI4-Lite register는 software 입장에서는 memory address처럼 보인다.
예를 들어 Vivado Address Editor에서 Systolic Array Controller에 base address를 `0xA000_0000`으로 배정했다고 하자.

그러면 controller 내부 register offset을 다음처럼 정할 수 있다.

| Offset | Register | 방향 | 의미 |
|---:|---|---|---|
| `0x00` | `CTRL` | W/R | start, done, busy |
| `0x04` | `M_SIZE` | W/R | output matrix row 수 |
| `0x08` | `N_SIZE` | W/R | output matrix column 수 |
| `0x0C` | `K_SIZE` | W/R | reduction dimension |
| `0x10` | `ACT_BASE` | W/R | Activation BRAM base word address |
| `0x14` | `WEIGHT_BASE` | W/R | Weight BRAM base word address |
| `0x18` | `ACC_BASE` | W/R | Accumulator BRAM base word address |
| `0x1C` | `STATUS` | R | done, busy, error |

그러면 PS software는 다음 주소에 접근하게 된다.

```text
0xA000_0000 + 0x00 -> CTRL
0xA000_0000 + 0x04 -> M_SIZE
0xA000_0000 + 0x08 -> N_SIZE
...
```

여기서 주의할 점은 base address의 의미다.
`ACT_BASE`, `WEIGHT_BASE`, `ACC_BASE`는 DDR byte address가 아니라 **BRAM word address**로 잡는 것이 자연스럽다.
Engine FSM이 BRAM을 word 단위로 읽기 때문이다.

반면 DMA register에 쓰는 source/destination address는 DDR의 byte address다.

```text
DMA source/destination address:
  DDR byte address

Systolic Array Engine base address:
  BRAM word address
```

이 차이를 분리해두면 software와 RTL 사이의 계약이 훨씬 명확해진다.

## Control Register 예시

가장 단순한 `CTRL` register는 다음처럼 둘 수 있다.

| Bit | 이름 | 방향 | 의미 |
|---:|---|---|---|
| 0 | `start` | W | 1을 쓰면 Engine 시작 |
| 1 | `done` | R | Engine 완료 |
| 2 | `busy` | R | Engine 동작 중 |
| 3 | `error` | R | 잘못된 설정 또는 내부 에러 |
| 4 | `irq_en` | W/R | interrupt enable |
| 5 | `done_clear` | W | done flag clear |

software가 실행하는 순서는 대략 다음과 같다.

```text
1. M_SIZE, N_SIZE, K_SIZE register write
2. ACT_BASE, WEIGHT_BASE, ACC_BASE register write
3. CTRL.start = 1 write
4. STATUS 또는 CTRL.done polling
5. done이 1이면 결과 확인
6. done_clear write
```

PYNQ에서 보면 이런 식의 코드가 된다.

```python
from pynq import MMIO

ctrl = MMIO(0xA0000000, 0x1000)

ctrl.write(0x04, 256)   # M_SIZE
ctrl.write(0x08, 256)   # N_SIZE
ctrl.write(0x0C, 1024)  # K_SIZE

ctrl.write(0x10, 0)     # ACT_BASE, BRAM word address
ctrl.write(0x14, 0)     # WEIGHT_BASE
ctrl.write(0x18, 0)     # ACC_BASE

ctrl.write(0x00, 0x1)   # start

while (ctrl.read(0x1C) & 0x1) == 0:
    pass
```

여기서 `MMIO.write()`는 결국 PS가 AXI4-Lite write transaction을 발생시키는 것이다.
PL 입장에서는 `S_AXI_LITE` slave로 들어온 write address와 write data를 보고 내부 register를 갱신한다.

## AXI4-Lite 신호

AXI4-Lite는 write와 read channel이 분리되어 있다.
처음 보면 신호가 많아 보이지만, 의미는 단순하다.

Write는 세 channel로 나뉜다.

| Channel | 주요 신호 | 의미 |
|---|---|---|
| Write Address | `AWADDR`, `AWVALID`, `AWREADY` | 어떤 register에 쓸지 |
| Write Data | `WDATA`, `WSTRB`, `WVALID`, `WREADY` | 어떤 값을 쓸지 |
| Write Response | `BRESP`, `BVALID`, `BREADY` | write 완료 응답 |

Read는 두 channel로 나뉜다.

| Channel | 주요 신호 | 의미 |
|---|---|---|
| Read Address | `ARADDR`, `ARVALID`, `ARREADY` | 어떤 register를 읽을지 |
| Read Data | `RDATA`, `RRESP`, `RVALID`, `RREADY` | 읽은 값과 응답 |

모든 channel은 `valid`와 `ready` handshake를 사용한다.
데이터나 주소는 `VALID && READY`가 동시에 1인 cycle에 전달된다.

예를 들어 write transaction은 개념적으로 다음 순서로 진행된다.

```text
PS -> PL: AWADDR = register offset, AWVALID = 1
PL -> PS: AWREADY = 1

PS -> PL: WDATA = write value, WVALID = 1
PL -> PS: WREADY = 1

PL -> PS: BVALID = 1
PS -> PL: BREADY = 1
```

Read transaction은 다음과 같다.

```text
PS -> PL: ARADDR = register offset, ARVALID = 1
PL -> PS: ARREADY = 1

PL -> PS: RDATA = register value, RVALID = 1
PS -> PL: RREADY = 1
```

실제 AXI4-Lite slave RTL을 직접 작성할 때는 이 handshake를 정확히 처리해야 한다.
하지만 Vivado에서 AXI4-Lite peripheral template을 만들거나 HLS/IP Packager를 쓰면 기본 register wrapper를 생성해준다.
처음에는 template을 사용하고, 내부 register와 Engine start/done만 연결하는 방식이 안전하다.

## DMA Register도 AXI4-Lite로 설정된다

앞 글에서 Activation DMA, Weight DMA, Result DMA를 따로 두었다.
이 DMA들도 모두 `S_AXI_LITE` control port를 가진다.

software는 AXI4-Lite를 통해 DMA register를 설정한다.

```text
Activation DMA S_AXI_LITE:
  MM2S source DDR address
  MM2S transfer length
  MM2S start

Weight DMA S_AXI_LITE:
  MM2S source DDR address
  MM2S transfer length
  MM2S start

Result DMA S_AXI_LITE:
  S2MM destination DDR address
  S2MM transfer length
  S2MM start
```

즉 AXI4-Lite는 custom controller뿐 아니라 Xilinx AXI DMA IP를 제어할 때도 쓰인다.
다만 DMA register map은 Xilinx IP가 이미 정의해둔 것이고, Systolic Array Controller register map은 우리가 직접 정의하는 것이다.

## 전체 실행 순서

이제 PS software 관점에서 전체 실행 순서를 다시 보면 다음과 같다.

```text
1. DDR에 Activation buffer 준비
2. DDR에 Weight buffer 준비

3. AXI4-Lite로 Activation DMA register 설정
4. AXI4-Lite로 Weight DMA register 설정
5. DMA가 DDR -> BRAM load 수행

6. AXI4-Lite로 Systolic Array Controller register 설정
7. AXI4-Lite로 start bit write
8. Engine이 BRAM에서 읽어 계산

9. AXI4-Lite로 Result DMA register 설정
10. DMA가 Accumulator BRAM -> DDR store 수행

11. PS가 DDR output buffer 확인
```

여기서 AXI4-Lite는 매 단계의 "명령"을 전달하고, DMA와 Engine은 실제 일을 수행한다.
software가 모든 데이터를 AXI4-Lite로 직접 옮기지 않는다는 점이 핵심이다.

## 정리

AXI4-Lite는 PS와 PL이 직접 통신하는 control path다.
PS는 AXI4-Lite master로 동작하고, PL의 DMA나 custom controller는 AXI4-Lite slave register를 제공한다.

DMA는 대량 데이터를 옮기고, AXI4-Lite는 그 DMA와 Systolic Array Engine을 설정한다.
이 둘을 분리해서 보면 전체 구조가 명확해진다.

```text
AXI4-Lite:
  register 설정, 상태 확인

AXI DMA + AXI-Stream:
  실제 matrix data 이동

Systolic Array Engine:
  BRAM에 준비된 data 계산
```

다음 단계에서는 이 AXI4-Lite register map을 기준으로 System Controller를 어떻게 만들지 정리할 수 있다.
