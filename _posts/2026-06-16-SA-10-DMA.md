## 개요

지금까지는 Systolic Array Engine 내부를 설계했다.
BRAM에 Activation과 Weight가 이미 저장되어 있다고 가정하고, Engine이 그 데이터를 읽어 GEMM을 수행한 뒤 결과를 다시 BRAM에 저장하는 구조를 만들었다.

하지만 실제 시스템에서는 입력 데이터가 처음부터 BRAM에 있는 것이 아니다.
보통 큰 행렬 데이터는 DDR에 있고, Systolic Array Engine은 PL 내부의 BRAM을 local buffer처럼 사용한다.

따라서 다음 문제가 생긴다.

```text
DDR에 있는 Activation/Weight를 누가 BRAM으로 옮길 것인가?
BRAM에 저장된 Output을 누가 다시 DDR로 옮길 것인가?
```

이 역할을 담당하는 블록이 DMA다.

DMA는 Direct Memory Access의 약자다.
CPU가 직접 데이터를 하나씩 읽고 쓰는 대신, DMA가 memory 간 데이터 이동을 수행한다.
CPU는 DMA에 source address, destination address, transfer size를 설정하고 시작 신호만 준다.
그 이후 실제 데이터 이동은 DMA가 처리한다.

## 왜 DMA가 필요한가?

Systolic Array는 많은 MAC 연산을 병렬로 수행한다.
하지만 계산 코어가 아무리 빨라도, 데이터를 제때 공급하지 못하면 전체 성능은 떨어진다.

예를 들어 CPU가 DDR에서 데이터를 읽어 BRAM에 직접 쓰는 방식을 생각해보자.
CPU는 다음 작업을 반복해야 한다.

```text
DDR read
-> bus transaction
-> BRAM write
-> address update
-> loop
```

이 방식은 제어가 단순해 보이지만, 큰 행렬을 다룰수록 CPU가 데이터 복사에 묶인다.
CPU는 원래 연산 설정, scheduling, interrupt 처리 같은 제어를 담당하는 쪽이 더 적합하다.
대량의 연속 데이터 이동은 DMA가 맡는 것이 자연스럽다.

DMA를 사용하면 역할이 다음처럼 나뉜다.

| 블록 | 역할 |
|---|---|
| CPU | 연산 설정, DMA 설정, 실행 시작 |
| DMA | DDR과 AXI-Stream 사이의 대량 데이터 이동 |
| BRAM | Systolic Array 가까이에 있는 local buffer |
| Systolic Array Engine | BRAM의 데이터를 읽어 GEMM 수행 |

즉 DMA는 계산을 빠르게 하는 블록이라기보다, 계산 코어가 데이터를 기다리지 않게 만드는 블록이다.

## 전체 데이터 이동 흐름

Systolic Array Engine을 시스템에 연결하면 전체 흐름은 다음과 같다.

```text
DDR
  -> Activation DMA
  -> AXI-Stream to BRAM Writer
  -> Activation BRAM

DDR
  -> Weight DMA
  -> AXI-Stream to BRAM Writer
  -> Weight BRAM

Activation BRAM / Weight BRAM
  -> Systolic Array Engine
  -> Accumulator BRAM

Accumulator BRAM
  -> BRAM to AXI-Stream Reader
  -> Result DMA
  -> DDR
```

앞에서는 DMA를 하나의 역할로 설명했지만, 실제 Block Design에서는 **Activation, Weight, Result마다 AXI DMA block을 하나씩 연결하는 구조**로 잡는다.
즉 총 3개의 AXI DMA IP를 둔다.

| DMA Block | 주로 사용하는 채널 | 데이터 방향 | 연결 대상 |
|---|---|---|---|
| Activation DMA | `MM2S` | DDR -> Stream | `axis_to_bram_writer` -> Activation BRAM |
| Weight DMA | `MM2S` | DDR -> Stream | `axis_to_bram_writer` -> Weight BRAM |
| Result DMA | `S2MM` | Stream -> DDR | Accumulator BRAM -> `bram_to_axis_reader` |

Activation과 Weight는 DDR에서 BRAM으로 데이터를 가져오는 입력 경로이므로 `MM2S` channel을 사용한다.
Result는 Accumulator BRAM의 데이터를 DDR로 내보내는 출력 경로이므로 `S2MM` channel을 사용한다.
AXI DMA IP는 read channel과 write channel을 모두 켤 수 있지만, 이 구조에서는 각 DMA의 사용 방향이 정해져 있으므로 쓰지 않는 channel은 꺼두면 Block Design이 더 단순해진다.
세 DMA의 memory-mapped AXI master port는 모두 PS DDR 쪽 AXI interconnect 또는 SmartConnect에 연결된다.

이를 순서로 쓰면 다음과 같다.

1. CPU가 DDR에 Activation과 Weight를 준비한다.
2. CPU가 Activation DMA에 Activation transfer를 설정한다.
3. Activation DMA의 MM2S channel이 DDR의 Activation을 AXI-Stream으로 내보낸다.
4. AXI-Stream to BRAM Writer가 stream data를 Activation BRAM에 저장한다.
5. CPU가 Weight DMA에 Weight transfer를 설정한다.
6. Weight DMA의 MM2S channel이 DDR의 Weight를 AXI-Stream으로 내보낸다.
7. AXI-Stream to BRAM Writer가 stream data를 Weight BRAM에 저장한다.
8. Controller가 Systolic Array Engine을 시작한다.
9. Engine이 BRAM에서 데이터를 읽어 GEMM을 수행한다.
10. Engine이 결과를 Accumulator BRAM에 저장한다.
11. BRAM to AXI-Stream Reader가 Accumulator BRAM의 결과를 stream으로 내보낸다.
12. Result DMA의 S2MM channel이 stream data를 DDR로 저장한다.
13. CPU가 DDR에 저장된 결과를 확인한다.

여기서 중요한 점은 DMA와 Systolic Array Engine이 같은 일을 하지 않는다는 것이다.
DMA는 memory와 stream 사이의 데이터 이동을 담당하고, Engine은 BRAM에 준비된 데이터를 계산한다.
Controller는 이 두 블록의 실행 순서를 맞춘다.

## AXI DMA Block

Vivado에서 `AXI Direct Memory Access` IP를 추가하면 다음과 같은 블록이 생긴다.

![](/assets/images/axi_dma_block.png)

이 블록의 핵심은 AXI Memory-Mapped interface와 AXI-Stream interface를 변환하는 것이다.
즉 AXI DMA는 단순히 DDR에서 BRAM으로 바로 복사하는 블록이 아니라, DDR 같은 memory-mapped 영역과 stream datapath 사이에서 데이터를 옮기는 블록이다.

포트를 역할별로 나누면 다음과 같다.

| 포트 | 방향 | 역할 |
|---|---|---|
| `S_AXI_LITE` | input/output | CPU가 DMA register를 설정하는 제어 포트 |
| `M_AXI_MM2S` | output | DDR에서 데이터를 읽기 위한 AXI master port |
| `M_AXIS_MM2S` | output | DDR에서 읽은 데이터를 AXI-Stream으로 내보내는 포트 |
| `S_AXIS_S2MM` | input | AXI-Stream 데이터를 DMA로 입력하는 포트 |
| `M_AXI_S2MM` | output | 입력받은 stream 데이터를 DDR에 쓰는 AXI master port |
| `mm2s_introut` | output | MM2S transfer 완료 또는 에러 interrupt |
| `s2mm_introut` | output | S2MM transfer 완료 또는 에러 interrupt |

이름은 조금 헷갈릴 수 있지만, 방향을 기준으로 보면 간단하다.

```text
MM2S: Memory-Mapped to Stream
  DDR -> AXI DMA -> AXI-Stream

S2MM: Stream to Memory-Mapped
  AXI-Stream -> AXI DMA -> DDR
```

따라서 Systolic Array 시스템에서 입력을 가져올 때는 `MM2S`를 사용한다.
반대로 계산 결과를 DDR에 저장할 때는 `S2MM`을 사용한다.

이번 설계에서는 이 방향을 DMA block 단위로 분리한다.
Activation DMA와 Weight DMA는 read channel만 사용하는 MM2S DMA로 두고, Result DMA는 write channel만 사용하는 S2MM DMA로 둔다.
이렇게 나누면 Activation load, Weight load, Result store의 register 설정과 interrupt를 독립적으로 볼 수 있어 디버깅이 쉽다.

## AXI DMA 설정

다음 이미지는 Activation DMA 또는 Weight DMA처럼 **DDR에서 BRAM으로 데이터를 가져오는 read용 DMA** 설정 화면이다.

![](/assets/images/axi_dma_read_config.png)

현재 설정에서는 Scatter Gather Engine을 사용하지 않고, 단순 register mode로 DMA를 제어한다.
처음 구현할 때는 이 방식이 더 이해하기 쉽다.

| 설정 | 의미 |
|---|---|
| `Enable Scatter Gather Engine` off | descriptor list 없이 register로 단일 transfer 제어 |
| `Enable Micro DMA` off | 일반 AXI DMA mode 사용 |
| `Enable Read Channel` on | MM2S channel 사용 |
| `Enable Write Channel` off | S2MM channel은 사용하지 않음 |
| `Memory Map Data Width` 128 | DDR 쪽 AXI read data width |
| `Stream Data Width` 128 | AXI-Stream output data width |
| `Max Burst Size` 128 | AXI memory read burst의 최대 길이 |
| `Width of Buffer Length Register` 14 | 한 번에 설정할 수 있는 transfer length 폭 |
| `Address Width` 32 | DMA가 사용하는 memory address 폭 |

여기서 read channel은 DDR에서 stream으로 데이터를 내보내는 channel이다.
write channel은 stream으로 들어온 데이터를 DDR에 쓰는 channel이다.

Activation/Weight BRAM은 128bit word를 사용하므로, read용 DMA의 `Stream Data Width`를 128bit로 맞추면 stream beat 하나가 BRAM word 하나와 바로 대응된다.
따라서 Activation DMA와 Weight DMA는 위 설정을 그대로 사용하면 된다.
Component Name은 캡처처럼 `axi_dma_0`로 두기보다 Block Design에서 구분하기 쉽게 `act_dma`, `weight_dma`처럼 바꾸는 편이 좋다.

다음 이미지는 **Accumulator BRAM의 결과를 DDR로 저장하는 Result DMA** 설정 화면이다.

![](/assets/images/axi_dma_write_config.png)

Result DMA는 stream으로 들어온 결과를 DDR에 쓰는 S2MM 경로만 사용한다.

| 설정 | 의미 |
|---|---|
| `Enable Scatter Gather Engine` off | descriptor list 없이 register로 단일 transfer 제어 |
| `Enable Micro DMA` off | 일반 AXI DMA mode 사용 |
| `Enable Read Channel` off | MM2S channel은 사용하지 않음 |
| `Enable Write Channel` on | S2MM channel 사용 |
| `Memory Map Data Width` 512 | DDR 쪽 AXI write data width |
| `Stream Data Width` 512 | AXI-Stream input data width |
| `Max Burst Size` 16 | AXI memory write burst의 최대 길이 |
| `Width of Buffer Length Register` 23 | transfer byte count register 폭 |
| `Address Width` 32 | DMA가 사용하는 memory address 폭 |

여기서 `Stream Data Width`를 512bit로 둔 이유는 Accumulator BRAM word가 512bit이기 때문이다.
`bram_to_axis_reader`도 `DATA_W = 512`로 만들면 Accumulator BRAM word 하나가 AXI-Stream beat 하나로 바로 나간다.
즉 512bit 결과 word를 128bit beat 4개로 쪼개는 width converter를 따로 두지 않아도 된다.

`Memory Map Data Width`도 512bit로 맞췄다.
이렇게 하면 DMA 내부에서는 stream 쪽 512bit beat와 memory-mapped write 쪽 512bit beat가 같은 폭으로 움직인다.
다만 실제 PS DDR 포트의 data width가 512bit가 아니면 Vivado의 SmartConnect나 interconnect가 width conversion을 수행한다.
따라서 Result DMA의 `M_AXI_S2MM`은 PS의 HP/HPC slave port 쪽 SmartConnect에 연결한다.

`Width of Buffer Length Register`는 23bit로 키웠다.
이 값은 한 번의 DMA transfer에서 설정할 수 있는 byte count의 폭이다.
예를 들어 512bit stream은 한 beat가 64byte이므로, `W512_D512` Accumulator BRAM 전체 512word를 옮기려면 다음 byte count가 필요하다.

```text
512 words * 64 bytes/word = 32768 bytes
```

기본처럼 14bit만 쓰면 최대 byte count가 약 16KB라서 512word 전체를 한 번에 보내기 부족하다.
23bit면 최대 약 8MB까지 한 번에 표현할 수 있으므로, 이후 matrix size가 커져도 여유가 있다.

세 개의 DMA block을 만들 때는 방향에 맞게 channel을 선택한다.

| DMA Block | Enable Read Channel | Enable Write Channel |
|---|---|---|
| Activation DMA | on | off |
| Weight DMA | on | off |
| Result DMA | off | on |

공통으로 `Enable Scatter Gather Engine`은 끄고, CPU가 AXI-Lite register를 직접 설정하는 단순 모드로 시작한다.
각 DMA는 자기 `S_AXI_LITE` 제어 포트를 갖기 때문에, CPU 입장에서는 Activation DMA register, Weight DMA register, Result DMA register를 각각 설정하는 구조가 된다.

Systolic Array와 연결할 때 가장 신경 써야 할 값은 `Stream Data Width`다.
이 width는 DMA가 한 cycle에 AXI-Stream으로 내보내거나 받아들이는 data width를 의미한다.

예를 들어 Stream Data Width가 32 bit라면, DMA의 stream data port는 32 bit 단위로 움직인다.
하지만 Systolic Array Engine의 BRAM port는 다음처럼 더 넓을 수 있다.

```verilog
input  logic [   ROWS*ACT_W-1:0] act_bram_data_i
input  logic [COLS*WEIGHT_W-1:0] weight_bram_data_i
output logic [   ROWS*ACC_W-1:0] acc_bram_data_o
```

따라서 DMA stream width와 BRAM word width가 다르면 width converter 또는 packing/unpacking logic이 필요하다.
처음에는 DMA stream width와 BRAM word width를 맞춰두는 것이 구현이 쉽다.

앞선 BRAM 설정을 기준으로 하면 Activation/Weight BRAM word는 128bit이고, Accumulator BRAM word는 512bit이다.
DMA Stream Data Width를 128bit로 두면 Activation/Weight 전송은 한 stream beat가 한 BRAM word와 바로 대응된다.
반면 Accumulator 결과는 512bit word로 저장하므로, Result DMA의 Stream Data Width를 512bit로 맞춘다.
이렇게 하면 Result 경로에서는 `bram_to_axis_reader`가 읽은 512bit BRAM word가 그대로 S2MM stream beat가 된다.

## DMA와 BRAM 연결

AXI DMA의 `M_AXIS_MM2S`를 BRAM에 바로 연결할 수는 없다.
`M_AXIS_MM2S`는 AXI-Stream이고, BRAM은 address/data/write enable 기반의 memory interface이기 때문이다.

따라서 입력 방향에는 stream을 받아 BRAM에 쓰는 모듈이 하나 필요하다.

```text
DDR
  -> M_AXI_MM2S
  -> Activation DMA / Weight DMA
  -> M_AXIS_MM2S
  -> axis_to_bram_writer
  -> Activation BRAM / Weight BRAM
```

`axis_to_bram_writer`는 다음 역할을 한다.

| 신호 | 역할 |
|---|---|
| `s_axis_tdata` | DMA에서 들어오는 data |
| `s_axis_tvalid` | DMA가 현재 data가 유효하다고 알림 |
| `s_axis_tready` | writer가 data를 받을 수 있다고 알림 |
| `s_axis_tlast` | transfer의 마지막 beat 표시 |
| `bram_we_o` | BRAM write enable |
| `bram_addr_o` | 현재 write address |
| `bram_data_o` | BRAM에 쓸 data |

동작은 단순하다.
`tvalid`와 `tready`가 동시에 1인 cycle마다 stream data 하나를 받고, BRAM address를 하나 증가시키며 저장한다.

```text
if tvalid && tready:
  BRAM[addr] = tdata
  addr = addr + 1
```

반대로 출력 방향에는 BRAM을 읽어 AXI-Stream으로 내보내는 모듈이 필요하다.

```text
Accumulator BRAM
  -> bram_to_axis_reader
  -> S_AXIS_S2MM
  -> Result DMA
  -> M_AXI_S2MM
  -> DDR
```

`bram_to_axis_reader`는 다음 역할을 한다.

| 신호 | 역할 |
|---|---|
| `bram_addr_o` | 읽을 BRAM address |
| `bram_data_i` | BRAM에서 읽은 data |
| `m_axis_tdata` | DMA로 보낼 stream data |
| `m_axis_tvalid` | reader가 현재 data가 유효하다고 알림 |
| `m_axis_tready` | DMA가 data를 받을 수 있다고 알림 |
| `m_axis_tlast` | 마지막 beat 표시 |

이 모듈도 `tvalid`와 `tready` handshake를 기준으로 움직인다.
DMA가 받을 준비가 되었을 때만 다음 BRAM word를 stream으로 내보내야 한다.

실제 구현은 submodule의 `external/SimpleSystolicArray/hdl/bram_to_axis_reader.sv`에 두었다.
이 모듈의 제어 인터페이스는 `axis_to_bram_writer`와 거의 대칭이다.

```verilog
input  logic              start_i,
input  logic [ADDR_W-1:0] base_addr_i,
input  logic [ LEN_W-1:0] length_i,
output logic              busy_o,
output logic              done_o,

output logic [DATA_W-1:0] m_axis_tdata_o,
output logic              m_axis_tvalid_o,
input  logic              m_axis_tready_i,
output logic              m_axis_tlast_o,

output logic              bram_en_o,
output logic [ADDR_W-1:0] bram_addr_o,
input  logic [DATA_W-1:0] bram_data_i
```

동작 순서는 다음과 같다.

```text
1. start_i가 들어오면 base_addr_i와 length_i를 latch한다.
2. bram_en_o와 bram_addr_o로 Accumulator BRAM read를 요청한다.
3. BRAM read latency 1 cycle 뒤 bram_data_i를 내부 register에 잡는다.
4. m_axis_tvalid_o를 1로 올리고 m_axis_tdata_o를 유지한다.
5. Result DMA가 m_axis_tready_i를 1로 올리면 해당 beat가 전송된다.
6. 마지막 beat에서는 m_axis_tlast_o를 1로 함께 내보낸다.
7. 마지막 beat handshake가 끝나면 done_o를 1 cycle pulse로 낸다.
```

여기서 중요한 점은 backpressure 처리다.
Result DMA가 잠시 받을 수 없어서 `m_axis_tready_i`를 0으로 두면 reader는 현재 `m_axis_tdata_o`와 `m_axis_tlast_o`를 그대로 유지한다.
즉 AXI-Stream 규칙대로 `tvalid && tready`가 동시에 1이 되는 cycle에만 다음 데이터로 넘어간다.

현재 구현은 이해하기 쉬운 **one outstanding read** 구조다.
BRAM read를 하나 요청하고, 그 데이터가 AXI-Stream으로 accept된 뒤 다음 BRAM read를 요청한다.
따라서 구조는 단순하지만, `tready`가 항상 1이어도 매 cycle 한 beat가 나가는 완전 pipelined reader는 아니다.
나중에 Result DMA 경로가 병목이 되면 prefetch register나 skid buffer를 추가해서 BRAM read와 AXI-Stream 전송을 겹치면 된다.

검증은 `external/SimpleSystolicArray/tb/tb_bram_to_axis_reader.sv`에서 진행했다.
테스트벤치는 1 cycle latency를 갖는 간단한 BRAM model을 두고 다음을 확인한다.

| 테스트 | 확인 내용 |
|---|---|
| reset | `busy_o`, `done_o`, `m_axis_tvalid_o`, `bram_en_o` 초기값 |
| zero length | `length_i == 0`일 때 즉시 `done_o` pulse |
| normal transfer | base address부터 순서대로 read하고 마지막 beat에서 `tlast` 출력 |
| backpressure | `tready`가 0이면 같은 data/last를 유지 |
| busy 중 start | transfer 중 새 start가 들어와도 기존 transfer 유지 |
| single beat | 길이 1인 transfer에서 첫 beat가 곧 `tlast` |

XSIM 단독 실행 결과는 다음과 같다.

```text
tb_bram_to_axis_reader PASS
```

## Systolic Array와의 연결 구조

Systolic Array Engine까지 포함하면 전체 연결은 다음처럼 볼 수 있다.

```text
                  control
CPU / PS --------------------------+
                                   |
                                   v
                              Controller
                                   |
        +----------+----------+-------------------------+
        |                     |                         |
        v                     v                         v
 Activation DMA MM2S   Weight DMA MM2S          Systolic Array Engine
        |                     |                         ^
        v                     v                         |
 axis_to_bram_writer   axis_to_bram_writer              |
        |                     |                         |
        v                     v                         |
 Activation BRAM       Weight BRAM ---------------------+

   Accumulator BRAM
              |
              v
      bram_to_axis_reader
              |
              v
        Result DMA S2MM
              |
              v
             DDR
```

Controller는 크게 세 종류의 블록을 제어한다.

| 블록 | Controller가 하는 일 |
|---|---|
| Activation/Weight/Result DMA | source/destination address, transfer length 설정 |
| AXI-Stream/BRAM adapter | 어느 BRAM에 쓸지, 시작 주소, 길이 설정 |
| Systolic Array Engine | matrix size, base address, start 설정 |

예를 들어 Activation을 로드할 때는 다음 순서가 된다.

1. Activation용 `axis_to_bram_writer`의 BRAM 시작 주소와 write length를 설정한다.
2. Activation DMA MM2S source address를 DDR Activation buffer로 설정한다.
3. Activation DMA MM2S transfer length를 설정하고 start한다.
4. Activation DMA가 `M_AXIS_MM2S`로 데이터를 내보낸다.
5. writer가 stream data를 Activation BRAM에 저장한다.
6. DMA interrupt 또는 done bit로 transfer 완료를 확인한다.

Weight도 같은 방식으로 로드한다.
다만 Weight DMA와 Weight용 writer를 사용하고, DDR source buffer도 Weight 쪽으로 바뀐다.

1. Weight용 `axis_to_bram_writer`의 BRAM 시작 주소와 write length를 설정한다.
2. Weight DMA MM2S source address를 DDR Weight buffer로 설정한다.
3. Weight DMA MM2S transfer length를 설정하고 start한다.
4. Weight DMA가 `M_AXIS_MM2S`로 데이터를 내보낸다.
5. writer가 stream data를 Weight BRAM에 저장한다.
6. DMA interrupt 또는 done bit로 transfer 완료를 확인한다.

Output을 저장할 때는 방향이 반대다.

1. `bram_to_axis_reader`의 source를 Accumulator BRAM으로 설정한다.
2. reader의 BRAM 시작 주소와 read length를 설정한다.
3. Result DMA S2MM destination address를 DDR Output buffer로 설정한다.
4. Result DMA S2MM transfer length를 설정하고 start한다.
5. reader가 Accumulator BRAM을 읽어 `S_AXIS_S2MM`으로 보낸다.
6. Result DMA가 stream data를 DDR에 저장한다.
7. DMA interrupt 또는 done bit로 transfer 완료를 확인한다.

이렇게 보면 AXI DMA는 Systolic Array Engine과 직접 계산 데이터를 주고받는 블록이 아니다.
DMA는 DDR과 stream 사이를 연결하고, stream과 BRAM 사이에는 별도의 adapter가 있으며, Engine은 BRAM만 바라본다.

## DMA가 보는 데이터

DMA는 행렬의 의미를 모른다.
DMA 입장에서 데이터는 단순한 byte stream이다.

```text
source address
destination address
transfer byte count
```

DMA는 이 세 가지를 기준으로 데이터를 옮긴다.
`A_{0,0}`인지, `W_{1,2}`인지, output tile인지 알 필요가 없다.

여기서 `transfer byte count`는 byte 단위다.
반면 BRAM adapter의 address는 보통 BRAM word 단위다.
예를 들어 Activation BRAM이 128bit라면 BRAM address 1 증가는 16 byte 이동과 같다.
따라서 DMA transfer length와 BRAM write length를 계산할 때는 다음 관계를 맞춰야 한다.

```text
bram_word_bytes = bram_word_width / 8
dma_byte_count  = bram_word_count * bram_word_bytes
```

반면 Systolic Array Engine은 BRAM에 저장된 데이터가 특정 memory layout을 따른다고 가정한다.
앞에서 정한 Activation/Weight layout과 다르게 저장되어 있으면, FSM이 올바른 주소를 생성하더라도 잘못된 데이터를 읽게 된다.

따라서 DMA 자체는 단순 복사만 수행하더라도, **DDR에 데이터를 배치하는 순서**는 Systolic Array의 memory layout과 맞아야 한다.

```text
CPU or software
  -> memory layout에 맞게 DDR buffer 구성
DMA
  -> DDR buffer를 AXI-Stream으로 전송
AXI-Stream/BRAM adapter
  -> stream data를 BRAM에 저장
Systolic Array Engine
  -> BRAM을 정해진 주소식으로 읽음
```

즉 memory layout은 DMA 내부 로직의 문제가 아니라, DMA가 옮기는 DDR buffer와 AXI-Stream/BRAM adapter가 BRAM에 쓰는 순서를 어떻게 맞출 것인가의 문제다.

## Activation 전송

Activation은 입력 행렬 A에 해당한다.
앞선 메모리 레이아웃에서 정한 순서대로 DDR buffer를 구성한 뒤, Activation DMA와 `axis_to_bram_writer`가 이를 Activation BRAM으로 옮긴다.

```text
DDR Activation Buffer
  -> Activation DMA
  -> axis_to_bram_writer
  -> Activation BRAM
```

Systolic Array Engine은 Activation BRAM의 base address를 기준으로 데이터를 읽는다.

```text
act_base_addr_i
```

즉 Activation DMA와 writer가 할 일은 `act_base_addr_i`부터 Engine이 읽을 수 있는 형태로 Activation 데이터를 채워두는 것이다.

여기서 주의할 점은 BRAM word width다.
예를 들어 Engine의 Activation BRAM port가 다음과 같다면,

```verilog
input logic [ROWS*ACT_W-1:0] act_bram_data_i
```

한 BRAM address에는 `ROWS`개의 activation lane이 packed되어 있다.
따라서 DMA가 stream beat 단위로 데이터를 옮기더라도, 최종적으로 BRAM에 들어가는 word는 Engine의 lane packing과 맞아야 한다.

## Weight 전송

Weight도 마찬가지다.
DDR의 Weight buffer는 Systolic Array Engine이 기대하는 weight memory layout으로 준비되어야 한다.

```text
DDR Weight Buffer
  -> Weight DMA
  -> axis_to_bram_writer
  -> Weight BRAM
```

Engine의 Weight BRAM port는 보통 다음처럼 `COLS`개 weight lane을 한 번에 읽는 형태다.

```verilog
input logic [COLS*WEIGHT_W-1:0] weight_bram_data_i
```

OS와 WS는 weight를 읽는 순서가 다를 수 있다.
특히 WS에서는 weight를 PE 내부에 load하는 과정 때문에 block 내부 read order를 신경 써야 한다.

하지만 DMA는 이 차이를 직접 이해하지 않는다.
OS용 Engine을 실행할 때는 OS memory layout으로 만든 Weight buffer를 옮기고, WS용 Engine을 실행할 때는 WS memory layout으로 만든 Weight buffer를 옮기면 된다.

```text
OS 실행:
  DDR weight buffer in OS layout
  -> Weight BRAM

WS 실행:
  DDR weight buffer in WS layout
  -> Weight BRAM
```

즉 dataflow별 memory layout 차이는 DMA 제어가 아니라, DMA transfer 전에 준비되는 buffer의 차이로 보는 것이 좋다.

## Output 전송

Systolic Array Engine의 결과는 Accumulator BRAM에 저장된다.
연산이 끝나면 Result DMA는 Accumulator BRAM의 데이터를 DDR로 옮긴다.

```text
Accumulator BRAM
  -> bram_to_axis_reader
  -> Result DMA
  -> DDR Output Buffer
```

이때도 DMA는 결과가 어떤 output tile인지 모른다.
그저 stream으로 들어온 데이터를 DDR의 output buffer로 저장한다.
Accumulator BRAM의 어느 주소를 어떤 순서로 읽을지는 `bram_to_axis_reader`가 결정한다.

따라서 software가 결과를 읽을 때는 Accumulator BRAM에 저장된 output layout을 알고 있어야 한다.
만약 output이 tile-major로 저장되어 있다면, DDR에 복사된 결과도 tile-major 순서가 된다.
필요하다면 software에서 이를 일반적인 row-major matrix로 다시 변환할 수 있다.

## Controller와 DMA의 관계

DMA는 단독으로 동작하지 않는다.
시스템 전체에서는 Controller가 DMA와 Systolic Array Engine의 실행 순서를 관리한다.

```text
IDLE
-> LOAD_ACT
-> LOAD_WEIGHT
-> RUN_ENGINE
-> STORE_OUTPUT
-> DONE
```

각 상태의 역할은 다음과 같다.

| 상태 | 동작 |
|---|---|
| `IDLE` | CPU의 start 대기 |
| `LOAD_ACT` | Activation DMA로 Activation을 BRAM에 적재 |
| `LOAD_WEIGHT` | Weight DMA로 Weight를 BRAM에 적재 |
| `RUN_ENGINE` | Systolic Array Engine 실행 |
| `STORE_OUTPUT` | Result DMA로 결과를 DDR에 저장 |
| `DONE` | CPU에 완료 알림 |

이 Controller는 Systolic Array Engine FSM과는 다른 레벨의 FSM이다.
Engine FSM은 BRAM 안에서 tile을 읽고 계산하는 순서를 제어한다.
반면 System Controller는 DMA와 Engine을 어떤 순서로 실행할지 제어한다.

```text
System Controller
  -> DMA 실행 순서 제어
  -> Systolic Array Engine start/done 제어

Systolic Array Engine FSM
  -> BRAM address 생성
  -> Loader/Array/Storer timing 제어
```

두 FSM의 책임을 분리하면 구조가 훨씬 명확해진다.

## Double Buffering으로 확장

가장 단순한 구조에서는 전체 Activation과 Weight를 BRAM에 모두 옮긴 뒤 Engine을 실행한다.
Engine이 끝나면 Output을 다시 DDR로 옮긴다.

```text
DMA load
-> compute
-> DMA store
```

하지만 이 방식은 DMA가 동작하는 동안 Engine이 쉬고, Engine이 동작하는 동안 DMA가 쉬는 구조가 된다.
성능을 더 높이려면 double buffering을 생각할 수 있다.

```text
Buffer 0 compute 중
Buffer 1 DMA load

Buffer 1 compute 중
Buffer 0 DMA load
```

이렇게 하면 다음 tile의 데이터를 옮기는 동안 현재 tile을 계산할 수 있다.
다만 double buffering을 적용하려면 BRAM을 두 벌로 나누거나, ping-pong buffer 구조를 만들어야 한다.
또한 Controller FSM도 더 복잡해진다.

따라서 처음에는 단순한 구조로 시작하는 것이 좋다.

```text
1. Activation load
2. Weight load
3. Engine run
4. Output store
```

이 구조가 동작한 뒤에 병목이 DMA transfer인지, compute인지 확인하고 double buffering을 추가해도 늦지 않다.

## 정리

Systolic Array Engine은 BRAM에 준비된 데이터를 계산한다.
AXI DMA는 DDR과 AXI-Stream 사이의 데이터를 옮기고, AXI-Stream/BRAM adapter가 stream data를 BRAM interface로 변환한다.
Controller는 DMA와 Engine의 실행 순서를 제어한다.

이 세 블록을 분리해서 보면 전체 시스템이 단순해진다.

| 블록 | 관심사 |
|---|---|
| DMA | 데이터를 어디서 어디로 얼마나 옮길 것인가 |
| Systolic Array Engine | BRAM 데이터를 어떤 순서로 읽어 GEMM을 수행할 것인가 |
| Controller | DMA와 Engine을 어떤 순서로 실행할 것인가 |
| Software | DDR buffer를 어떤 memory layout으로 준비할 것인가 |

결국 DMA를 붙인다는 것은 단순히 복사 모듈을 추가하는 것이 아니다.
Systolic Array Engine이 기대하는 memory layout과, DDR buffer를 준비하는 software, 그리고 DMA/Engine을 순서대로 실행하는 Controller가 함께 맞아야 한다.

다음 단계에서는 이 흐름을 기준으로 실제 Controller를 설계할 수 있다.
Controller는 CPU가 준 설정값을 받아 DMA load, Engine run, DMA store를 순서대로 실행하는 top-level 제어 블록이 된다.
